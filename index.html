Lab challenges

writeups structured for coursework

Lab 1 — Insecure direct object reference (IDOR) in a web API

Problem statement

A training REST API returned user profile JSON at /api/users/{id}. Authenticated as User A, changing {id} exposed User B’s PII without authorization checks.

Approach

Mapped endpoints with Burp Suite, replayed requests with modified IDs, confirmed consistent leakage. Documented reproduction steps and proposed server-side authorization tied to session subject.

Tools used

Burp Suite Community, browser devtools, notes in Markdown.

Key lessons learned

  • Never trust client-supplied identifiers alone—enforce resource ownership on every request.
  • Automated integration tests should cover horizontal privilege escalation cases.

Lab 2 — Analysing TLS and DNS in a captive portal scenario (Wireshark)

Problem statement

Training exercise: given a PCAP from a lab network, identify DNS queries, TLS ClientHello patterns, and a suspicious cleartext HTTP redirect.

Approach

Applied display filters (dns, tls, http), followed TCP streams, correlated timestamps with a provided incident timeline.

Tools used

Wireshark, optional tshark for one-liner exports.

Key lessons learned

  • Cleartext control-plane traffic remains a common weak link beside “strong” app crypto.
  • Structured PCAP notes speed up peer review and grading.

Lab 3 — Linux privilege escalation checklist (training VM)

Problem statement

VM with low-privilege shell; objective to obtain root and capture the proof flag in /root.

Approach

Ran LinPEAS-style enumeration mindset manually: SUID binaries, cron jobs, weak sudo entries, world-writable paths. Chained a misconfigured binary to elevated shell.

Tools used

SSH, find, grep, sudo -l, optional LinPEAS script (where permitted by course policy).

Key lessons learned

  • Post-exploitation is often “boring” misconfigurations, not exotic zero-days.
  • Always verify scope and rules of engagement before running aggressive scanners.
